Children

Los hijos en ReasonReact son como los hijos en ReactJS, excepto que ofrecemos nuevas funcionalidades aprovechando las ventajas que ofrece lenguaje.

Imaginemos que la gente está usando tu componente de la siguiente forma <Animation><div /></Animation>. En el componente Animation, querrías limitar que el número de children que recibe (en este caso ><div />) sea sea solamente uno. Es decir, te gustaría lanzar un error en el caso de: <Animation><div /><div /></Animation>.

ReactJS ofrece una función para ello:

// en Animation.js
render: () => {
  return React.Children.only(this.children);
}

O quizá hayas create un componente Layout que acepta exactamente dos hijos (e.g. <Layout><MyColumn /><MyColumn /></Layout>):

// Layout.js
render: () => {
  if (React.Children.count(this.props.children) !== 2) {
    // ... error
  }
  return (
    <div>
      {this.props.children[0]}
      <Separator />
      {this.props.children[1]}
    </div>
  );
}

O a lo mejor requieres un callback hijo:

<br />render: () => {
  return React.Children.only(this.props.children('hello'));
}

Como autor de estos componentes JS, sólo puedes tener la esperanza de que el usuario haya pasado los hijos que toca a tu componente, o lanzar un error en caso contrario. En ReasonReact, la prop children está tipada y previene el uso incorrecto por parte del consumidor. Es una conclusión natural en el momento en que te das cuenta de que la prop children ¡es básicamente igual que cualquier otra prop! ¡En efecto, puedes pasar una tupla, variant, record, etc. como children! ¡Esto significa también que no tenemos que implementar el equivalente a las utilidades React.Children.* en ReasonReact en absoluto!

Sin embargo, debido a razones sintácticas y de interoperabilidad, hemos puesto algunas restricciones sobre el uso de children:

  • Elementos del DOM como div, span y otros requieren que children sea de tipo array(reactElement). Por ejemplo, <div> [1, 2, 3] </div> o <span> Some(10) </span> no funcionarían, ya que list y option serían pasados directamente a los div/span de ReactJS y esos tipos de datos no tienen significado en JS.
  • Los elementos definidos por el usuario, por defecto, también aceptan un array (de cualquier contenido). Esto es debido a una limitación tonta de la sintaxis de JSX, descrita a continuación.

Limitación Sintáctica

Técnicamente, si has escrito un component que acepta una tupla como children:

/* MyForm.re */
type tuple2Children = (ReasonReact.reactElement, ReasonReact.reactElement);

let make = (children: tuple2Children) => {
  ...component,
  render: _self => {
    <div>
      {fst(children)}
      <Separator />
      {snd(children)}
    </div>
  }
};

Then you can expect the user to pass you a tuple:

<MyForm> (<div />, <div />) </MyForm>

Sin embargo esto daría un error:

This has type:
  array('a)
But somewhere wanted:
  tuple2Children (defined as (ReasonReact.reactElement, ReasonReact.reactElement))

Significa que MyForm espera una tupla, ¡pero le estás pasando un array en su lugar! Qué está ocurriendo? Bueno, si observamos qué ocurre durante la transformación de JSX vemos que:

<MyLayout> a b </MyLayout>
<MyLayout> a </MyLayout>

Se transforma en realidad en:

ReasonReact.element(
  MyLayout.make([|a, b|])
);
ReasonReact.element(
  MyLayout.make([|a|])
);

¿Ves como los children del segundo MyLayout también están dentro de un array? No podemos especializar el caso de <MyLayout> a </MyLayout> y pasar a sin estar dentro de un array (eso llevaría a crear errores incluso más confusos). Ya que children suele ser un array. Hemos decidido pasarlo siempre como array, independientemente de la cantidad de items que haya el el JSX. Pero, ¿qué ocurre si de verdad quieres pasar datos sin estar dentro de un array?

Expansión de hijos

Simplemente usa <MyLayout> ...a </MyLayout>. Este código será transformado a ReasonReact.element(MyLayout.make(a)), sin el array.

Consejos y sugerencias

A continuación se muestran algunos casos de uso para la expansión de hijos ...a junto a las estructuras de datos incluidas en Reason:

  • Un componente layout que requiere una tupla de 2 elementos React y los muestra adyacentemente:

    <MyForm> ...(<div />, <div />) </MyForm>

  • Un componente que requiere un callback, sin necesidad de usar React.Children.only:

    <Motion> ...((name) => <div className=name />) </Motion>

  • Una función de utilidad de layout que acepta una variante de TwoRows | ThreeRows | FourColumns

    <Layout> ...(ThreeRows(<div />, child2, child3)) </Layout>

Excepciones

Siguiendo el razonamiento de la sección anterior, este código debería funcionar:

let children = [| <div /> |];
<div> ...children </div>; /* <--- this line */

En realidad, sólo funciona con componentes de usuario (que empiezan en mayúscula), pero no funciona con componentes DOM (en minúscula). Esto es debido a alguna limitaciones relacionadas con los bindings; ¡lo mejoraremos en un futuro! Mientras tanto, para pasar children a un elemento del DOM, e.g. <div className=bar> ...children </div> usa lo siguiente:

ReasonReact.createDomElement("div", ~props={"className": bar}, children);
