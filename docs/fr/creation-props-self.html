Creation, Props & Self

Cette documentation nécessite une connaissance relative de ReactJS.

ReasonReact n'utilise pas/n'a pas besoin de classes. L'API de création de composants retourne un simple "record", dont vous pouvez surcharger les champs (par exemple render).

Le modèle de composant est créé par le biais de ReasonReact.statelessComponent("TheComponentName"). The string being passed is for debugging purposes (the equivalent of ReactJS' displayName).

À titre d’exemple, voici le fichier Greeting.re :

let component = ReasonReact.statelessComponent("Greeting");

En ReactJS, vous auriez créé un composant en tant que classe et l'auriez appelé à travers du JSX qui le transforme en React.createElement(myClass, {prop1: 'hello'}) en interne. En ReasonReact, au lieu de passer la 'classe' complète (ie modèle de composant) dans une hypothétique fonction ReasonReact.createElement, nous allons plutôt définir une fonction make :

/* still in Greeting.re */
let component = ReasonReact.statelessComponent("Greeting");

let make = (~name, _children) => {
  ...component, /* spread the template's other defaults into here  */
  render: _self => <div> {ReasonReact.string(name)} </div>
};

La fonction make est la fonction appelée par le JSX de ReasonReact, décrite plus tard. Pour le moment, la manière non-JSX d'appeler et de faire le rendu d'un composant est :

ReasonReact.element(Greeting.make(~name="John", [||])) /* la fonction `make` du module `Greeting` */
/* équivaut à <Greeting name="John" /> */

Il est nécessaire que make retourne le 'record' de composant défini ci dessus. Vous pouvez surcharger quelques champs, tels que les familiers render, initialState, didMount, etc., tous décrits plus loin.

Note: ne pas inliner let component dans le corps de la fonction make comme ce qui suit !

let make = _children => {
  ...(ReasonReact.statelessComponent("Greeting")),
  render: self => blabla
}

Puisque make est appelé à chaque invocation JSX, vous créériez accidentellement un nouveau composant à chaque fois.

Propriétés

Les propriétés sont juste des arguments nommés de la fonction make, vue plus haut. Ils peuvent également être facultatifs et/ou avoir des valeurs par défaut, exemple: let make = (~name, ~age=?, ~className="box", _children) => ....

La dernière propriété doit être children. Si vous ne l’utilisez pas, ignorez la simplement en la nommant _ ou _children. Les noms commençant par 'underscore' ne déclenchent pas d'alertes lorsqu'ils sont inutilisés.

Le nom d'une propriété ne peut pas être ref ni key. Ce sont des noms réservés, tout comme en ReactJS.

Selon cet exemple, vous appelleriez ce composant depuis un autre fichier par le biais de <Foo name="Reason" />. Si className est omis, sa valeur par défaut est "box". La valeur par défaut de age est None. Si vous souhaitez passer explicitement age, il suffit de faire : <Foo name="Reason" age=20 />.

Astuces concernant la propagation des propriétés

Parfois en ReactJS, vous recevez une propriété du parent que vous souhaiteriez transmettre directement à l’enfant :

<Foo name="Reason" age={this.props.age} />

C’est une source d'erreurs, parce que this.props.age pourrait être accidentellement changé en un nombre nullable alors que Foo n’attend pas qu’il le soit, ou vice versa ; il pourrait être nullable avant, et maintenant qu'il ne l'est plus Foo se retrouve avec un contrôle de nullité inutile quelque part dans la fonction de rendu.

En Reason, si vous voulez passer de façon explicite une propriété ageFromProps optionelle (dont le type est option int, c'est à dire None | Some int), ce qui suit ne fonctionnerait pas:

<Foo name="Reason" age=ageFromProps />

Tout simplement parce que age attend un int normal lorsqu'on appelle Foo avec, non pas un option int ! Naivement, vous pourriez résoudre le problème ainsi :

switch (ageFromProps) {
| None => <Foo name="Reason" />
| Some(nonNullableAge) => <Foo name="Reason" age=nonNullableAge />
}

Pénible. Heureusement, il y a une meilleure façon de passer explicitement une valeur optionnelle :

<Foo name="Reason" age=?ageFromProps />

Cela se traduit par "Je comprend que age est optionel et que lorsque j'utilise le label je dois passer un entier. Mais je tiens à transmettre une valeur option explicitement". Ce n’est pas un truc JSX que nous avons créé ; cela fait juste partie du langage ! See the section on "Explicitly Passed Optional" in the Reason docs.

self

Vous avez certainement vu le code render: (self) => ... dans make. Le concept JavaScript this n'existe pas en ReasonReact (bien qu'il puisse exister en Reason, puisqu'il a un système objet optionnel) ; l'équivalence de this s'appelle self. It's a record that contains state, handle and send, which we pass around to the lifecycle events, render and a few others, when they need the bag of information. Ces concepts seront expliqués plus tard.
